    <h2 id="installation-steps-windows-scripts">Installation Steps</h2>

    1. **Extract the Zip Archive**

    Unzip the downloaded file into the directory where you want to install OIBus (e.g., `C:\Program Files\OIBus`).

    2. **Run the Installation Script**
    Open an **administrator terminal** in the extracted folder and run:
    ``` commandline title="Usage"
    install.bat -c "[your-data-folder-path]"
    ```

    | Flag | Description | Default |
    |------|-------------|---------|
    | `-c` | Path to the OIBus data folder (cache, logs, configuration) | prompted if omitted |
    | `-n` | Windows service name — use a unique name to run multiple OIBus instances on the same machine | `OIBus` |
    | `-u` | Admin username written to `oibus.init.json` on first install | omitted (OIBus default applies) |
    | `-p` | Admin password written to `oibus.init.json` on first install | omitted (OIBus default applies) |
    | `-port` | Port OIBus listens on, written to `oibus.init.json` on first install | omitted (OIBus default applies) |
    | `-su` | Windows account the OIBus service runs as (e.g. `DOMAIN\oibus-svc`) — required for network share access | omitted (LocalSystem) |
    | `-sp` | Password for the service account specified with `-su` | omitted |

    :::info
    `-u`, `-p`, and `-port` are only used when `oibus.db` does not yet exist (i.e. on the very first install).
    They are written to an `oibus.init.json` file in the data folder and applied by OIBus on first startup.
    Omitted flags are not written to the file — OIBus will use its built-in defaults for those values.
    On subsequent installs or updates the file is not recreated, so these flags have no effect.
    :::

    :::tip
    - If you omit the `-c` argument, the script will prompt you for the data path.
    - For data safety, consider placing the data folder on a **separate disk** (e.g., `D:\OIBusData`). This helps
    prevent disk space issues on the system drive and improves reliability.
    :::

    **Example:**
    ``` commandline title="Example with terminal outputs"

    > The "OIBus" service has been successfully installed!

    > nssm.exe stop OIBus

    ```

    Proceed to the [first access guide](./first-access.mdx) to configure OIBus.

    <h2 id="update-oibus-windows-scripts">Update OIBus</h2>

    4. Restart the OIBus service.
    5. On first startup, the `oibus.db` configuration database will be automatically upgraded.


    The data folder is **not** automatically removed. Delete it manually if needed.

## Network share access (UNC paths)

By default, the OIBus Windows service runs under the **LocalSystem** account. LocalSystem has no domain or
network credentials, which means it **cannot access UNC paths** such as `\\server\share\data`. Attempting to
use a network path in a connector (e.g. the Folder Scanner input folder or the File Writer output folder)
will fail with an access-denied or path-not-found error.

There are two ways to fix this:

### Option 1 — Configure a service account (recommended)

Change the Windows service logon account to a domain or local user that already has the required network
permissions. This is the cleanest solution and works for all connectors and all UNC paths automatically.

- **GUI installer**: enter the account on the *Windows Service Account* page during installation.
- **Script install**: pass the account with `-su` and `-sp`:

```commandline
install.bat -c "D:\OIBusData" -su "DOMAIN\oibus-svc" -sp "SecretPassword"
```

- **After install (manual)**: use the Windows Service Manager or NSSM to change the service logon account:

```commandline
nssm.exe set OIBus ObjectName "DOMAIN\oibus-svc" "SecretPassword"
```

### Option 2 — SMB credentials in the connector settings

If changing the service account is not possible, you can provide SMB credentials directly in the
**Folder Scanner** (south connector) or **File Writer** (north connector) settings. OIBus will call
`net use` to authenticate against the UNC path before each operation.

Fill in these optional fields in the connector settings:

| Field | Description |
|-------|-------------|
| SMB username | Windows account with access to the share |
| SMB password | Password for the account |
| SMB domain | Active Directory domain (leave empty for local accounts) |

:::caution Windows only
The SMB credential fields have no effect on Linux or macOS. On those platforms, mount the network share
through the OS (e.g. `/etc/fstab` with `cifs`) and point the connector at the resulting mount point.
:::
